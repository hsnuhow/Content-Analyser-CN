# 部署鐵則 / DEPLOYMENT IRON RULE

**狀態：不可移除、不可弱化。** 此文件記錄一條絕對規則，凌駕一切便利性考量。

---

## 鐵則

> **任何 build 或 deploy 操作，在收到使用者明確的「部署口令」之前，絕對禁止執行。**

這包含但不限於：
- `gcloud builds submit ...`
- `gcloud run deploy ...`
- `bash deploy.sh`
- `firebase deploy ...`
- 任何重新部署、單一服務部署、背景部署

## 為什麼

部署會產生實際的雲端資源、費用與線上狀態變更。這是不可逆、有副作用的操作，
**決定權永遠在使用者手上**，不在 AI 手上。

## 絕對禁止的行為

- ❌ 完成程式碼後，因為「邏輯上下一步就是部署」就自行部署。
- ❌ 把「核准開發」當成「也可以部署」。**核准開發 ≠ 核准部署。**
- ❌ 先啟動部署、再請使用者補口令（先斬後奏）。
- ❌ 以「只是測試」「只是重新部署」「只是修正」為理由跳過口令。

## 正確流程

1. 完成程式碼修改、語法驗證、commit、push。
2. **停下來。** 明確告訴使用者「已就緒，是否部署？」
3. 等待使用者輸入有效部署口令。
4. 收到口令後，才執行 build / deploy。

## 有效部署口令（只有這些才算授權）

| 口令 | 動作 |
|------|------|
| `核准部署` | 部署三個 Cloud Run 服務（等同「核准部署：正式」）|
| `核准部署：正式` | 部署至生產環境 |
| `核准部署：測試` | 部署測試 |
| `核准部署：單一服務` | 只部署指定的單一服務 |

「可以」「好」「繼續」「OK」「核准開發」「核准修正」**都不是**部署授權。

## 違反的後果

違反此鐵則屬於嚴重操作錯誤。一旦發生：立即停止、承認錯誤、不得再犯。
